
Slack 实时消息系统的关键设计围绕以下几个核心问题：
- 消息如何通过 WebSocket 在全球范围内 500ms 内到达用户
- 如何用一致性哈希管理数以千万计的 Channel
- Pub/Sub 如何在 Gateway Server 和 Channel Server 之间协调

以下是对这套系统设计的文字解读，供你在面试中系统化陈述：

**核心架构三层分离**

Slack 的后端由三类服务器组成。Channel Server（CS）是有状态的内存服务器，每台在峰值时服务约 1600 万个频道，通过一致性哈希将频道映射到具体节点。Gateway Server（GS）同样是有状态的内存服务器，负责维护用户信息和 WebSocket 订阅关系，是客户端与 CS 之间的接口。Admin Server（AS）是无状态的，作为 Webapp 和 CS 之间的路由中间层。

**消息发送路径的关键决策**

客户端通过 HTTP POST 发送消息到 Webapp，Webapp 先记录日志持久化并建立索引，然后再调用 Channel Server 进行实时广播。这种方式改善了崩溃安全性，并确保消息要么被存储，要么发送方收到明确的失败通知。移动端客户端也因此不需要建立持久 WebSocket 连接即可发送消息。

**WebSocket 连接建立流程**

客户端启动时，先从 Webapp 获取用户 token 和 WebSocket 连接配置，然后连接到最近的边缘节点，Envoy 将请求转发给 GS。GS 获取该用户所在的所有频道信息后，通过一致性哈希异步订阅对应的 CS。

**Flannel 解决启动延迟**

Slack 引入了 Flannel——一个地理分布式微服务，维护预热的内存团队元数据缓存，监听实时事件保持缓存更新，从区域副本本地提供会话数据，将会话启动从一个计算密集型操作转变为缓存驱动的过程。P99 延迟因此从 2000ms 降至 200ms。

**去重与消息顺序**

他们对消息进行 salt 处理以防止同一条消息展示两次——salt 是一个随机唯一 token。消息顺序则由 CS 在广播前全局定序，保证所有用户看到一致的对话历史。

**一致性哈希的完整工作原理**

![一致性哈希](../assets/html/consistent_hash_ring_interactive.html "一致性哈希")

**哈希环的构造。** 把 0°~360° 想象成一个圆环。Slack 把每个 Channel Server 的标识符（通常是 IP + 端口）做哈希，得到一个 0~360 之间的数字，然后把这台服务器"钉"在环上对应的位置。5 台服务器就有 5 个点均匀（或近似均匀）分布在环上。

**路由一条消息。** 当 Admin Server 收到一条发往 `#general` 频道的消息，它对 `channel_id` 做同样的哈希，得到一个角度，比如 73°。然后规则很简单：顺时针走，碰到的第一台服务器就是目标 CS。73° 顺时针下去，遇到 CS-2（102°），所以 `#general` 永远由 CS-2 处理。

**为什么好用。** 普通取模（`hash % N`）的问题是：只要 N 变化（任何一台服务器宕机或扩容），几乎所有频道都要重新映射。一致性哈希的巧妙之处在于：当某台 CS 宕机时，只有它"负责"那段弧上的频道需要重新迁移到下一台，其他频道完全不受影响。

**CHARM 的角色。** Slack 用 CHARM（Consistent Hash Ring Manager）来管理这个环。它持续监控所有 CS 的健康状态，一旦某台宕机，它会在 Consul（服务注册中心）里更新哈希环配置，新配置自动推送给所有 AS 和 CS。整个替换过程 20 秒内完成，期间只有映射到宕机节点那段弧的频道会有短暂延迟，其余频道完全无感知。

**Admin Server 本身是无状态的。** 它不缓存任何路由表，每次路由都从 Consul 拉最新的哈希环配置，或者从本地内存里的最近副本计算。这样 AS 可以任意扩容，不存在状态同步问题。路由逻辑就是一次哈希计算加一次顺时针查找，极其轻量。
这是整套架构里最精妙的一环——CS 本身不"知道"用户在哪，它依赖的是 GS 主动来**订阅**。这是一个经典的 Pub/Sub 模式。可以依次点击四个按钮，完整感受订阅、广播、新用户加入、用户离线的全过程。

**Channel Server 如何知道该把消息发给哪些 Gateway Server ,机制的核心：是 GS 找 CS，不是 CS 找 GS**

CS 自己从不主动发现任何 GS。整个系统的设计是**反过来**的：

**订阅建立。** 当一个用户连上 GS 并打开 `#general` 这个频道时，GS 会做一次一致性哈希（hash(`#general`) → CS-3），然后主动向 CS-3 发一个订阅请求：「我这里有用户在看 #general，有消息请通知我」。CS-3 于是把这个 GS 的地址加入 `#general` 的订阅列表。

**广播触发。** 当有消息发到 CS-3 时，它只需要遍历 `#general` 的订阅列表，逐一推送给每个 GS。CS 不关心用户在哪，只关心「哪些 GS 订阅了这个 channel」。每个 GS 收到后，再从自己的内存映射里找到所有监听这个 channel 的 WebSocket 连接，推送过去。

**订阅撤销。** 当一个 GS 上该 channel 的最后一个用户下线，GS 会向 CS 发取消订阅。CS 把这个 GS 从列表里移除——下次广播就不再通知它了。这就是为什么 GS-SA 没有用户时不会收到任何消息，节省了跨洲际的无效网络传输。

**为什么这个设计很漂亮。** CS 是纯粹的 Pub/Sub broker，它的订阅列表完全由 GS 的用户实际连接情况驱动，实时动态更新。GS 数量可以任意横向扩展，CS 根本不需要知道全局有多少 GS——它只需要维护好「谁订阅了我」这张动态名单。这也是为什么 Slack 能同时维持数千万 WebSocket 连接，却不会在 CS 层造成瓶颈。
