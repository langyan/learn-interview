### 1. HashMap JDK8 底层原理 & 线程不安全

**底层结构：** 数组 + 链表 + 红黑树

- 默认容量 16，负载因子 0.75
- 链表长度 ≥ 8 且数组长度 ≥ 64 时，链表转红黑树
- 扩容时容量翻倍，重新 rehash

**为什么线程不安全：**
- `put` 操作没有加锁，多线程并发写会导致**数据覆盖**
- 扩容时多线程操作可能导致**死循环**（JDK7 头插法，JDK8 尾插法改善但仍不安全）
- 推荐使用 `ConcurrentHashMap`（分段锁 / CAS + synchronized）

### 2. synchronized 锁升级 & ReentrantLock 选型

**锁升级过程（单向，不可降级）：**
```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
```
- **偏向锁：** 只有一个线程访问，Mark Word 记录线程ID
- **轻量级锁：** 多线程竞争但不激烈，CAS 自旋
- **重量级锁：** 竞争激烈，挂起线程，OS 级别互斥

**支付场景选型：**

| 场景 | 推荐 |
|------|------|
| 简单同步、代码块短 | synchronized |
| 需要超时、可中断、公平锁 | ReentrantLock |
| 支付扣款（需要超时防死锁） | **ReentrantLock** `tryLock(timeout)` |

### 3. JVM 内存结构 & FullGC 排查

**JVM 内存区域：**
- **堆（Heap）：** 对象存储，Young（Eden+S0+S1）+ Old
- **方法区/元空间：** 类信息、常量池
- **虚拟机栈：** 线程私有，栈帧
- **本地方法栈 / 程序计数器**

**线上频繁 FullGC 排查步骤：**
```bash
1. jstat -gcutil <pid> 1000   # 查看各区GC频率
2. jmap -heap <pid>            # 查看堆内存使用
3. jmap -histo <pid>           # 查看对象分布
4. 导出 dump: jmap -dump:format=b,file=heap.hprof <pid>
5. MAT/VisualVM 分析大对象、内存泄漏
```
**常见原因：** 大对象直接进老年代、内存泄漏、元空间不足、显式 `System.gc()`

### 4. InnoDB 为什么用 B+ 树 & 最左匹配失效

**为什么用 B+ 树：**
- 非叶节点只存 key，层高低（通常 3-4 层），**减少磁盘 IO**
- 叶节点双向链表，**范围查询高效**
- 相比 B 树，数据全在叶子，**扫表更稳定**

**最左匹配失效场景：**
```sql
-- 联合索引 (a, b, c)
SELECT * FROM t WHERE b = 1            -- ❌ 跳过a，失效
SELECT * FROM t WHERE a = 1 AND c = 1  -- ⚠️ 只用a，c失效
SELECT * FROM t WHERE a LIKE '%xxx'    -- ❌ 左模糊，失效
SELECT * FROM t WHERE a + 1 = 2       -- ❌ 函数/运算，失效
SELECT * FROM t WHERE a = 1 AND b > 1 AND c = 1 -- ⚠️ c失效（b范围后停止）
```

### 5. MySQL MVCC & undo log / read view

**MVCC 实现原理：**
- 每行数据有隐藏字段：`trx_id`（事务ID）、`roll_pointer`（回滚指针）
- 通过 undo log 形成**版本链**

**undo log 作用：**
- 记录数据修改前的值，用于**事务回滚**
- 为 MVCC 提供历史版本数据

**read view 作用：**
- 快照读时生成，记录当前活跃事务列表
- 判断某版本数据对当前事务**是否可见**
- RC（每次读生成新 read view） vs RR（事务首次读生成，不变）

### 6. Redis 缓存穿透、击穿、雪崩解决方案

| 问题 | 原因 | 级解决方案 |
|------|------|--------------|
| **穿透** | 查询不存在的key | ① 布隆过滤器拦截；② 空值缓存（TTL短） |
| **击穿** | 热点key过期，大量请求打DB | ① 互斥锁（只1个线程重建）；② 逻辑过期（不设TTL，异步更新） |
| **雪崩** | 大量key同时过期 | ① TTL加随机值；② 多级缓存；③ 熔断降级；④ Redis集群高可用 |

### 7. Redis 分布式锁缺陷 & 主从切换失效

**普通 SETNX 锁缺陷：**
- 主节点加锁成功，**未同步到从节点时主节点宕机**
- 从节点升为主节点，其他线程可再次加锁 → **锁失效**

**解决方案：**

```
方案1：Redlock（红锁）
- 向 N 个独立 Redis 节点加锁
- 超过半数（N/2+1）加锁成功才算成功
- 缺点：性能差、Martin Kleppmann 指出仍有缺陷

方案2：Zookeeper 分布式锁
- 基于临时顺序节点，天然解决主从问题

方案3：业务层幂等
- 依赖DB唯一约束/版本号做最终保障
```

### 8. Spring 事务传播机制 & @Transactional 失效场景

**7种传播机制：**

| 传播行为 | 说明 |
|---------|------|
| REQUIRED | 有事务加入，无则新建（默认） |
| REQUIRES_NEW | 总是新建，挂起原事务 |
| SUPPORTS | 有事务加入，无则非事务运行 |
| NOT_SUPPORTED | 非事务运行，挂起原事务 |
| MANDATORY | 必须在事务中，否则抛异常 |
| NEVER | 不能在事务中 |
| NESTED | 嵌套事务（保存点） |

**@Transactional 常见失效场景：**
```java
1. 同类内部调用（this.方法()）→ 绕过代理
2. 方法非 public
3. 异常被 catch 吞掉，未抛出
4. 抛出的是受检异常（默认只回滚 RuntimeException）
5. 数据库引擎不支持事务（如 MyISAM）
6. 多线程场景（事务绑定在 ThreadLocal）
```

### 9. 线程池 7 大参数 & 避免 OOM

**7大参数：**
```java
new ThreadPoolExecutor(
    corePoolSize,      // 核心线程数
    maximumPoolSize,   // 最大线程数
    keepAliveTime,     // 空闲线程存活时间
    TimeUnit.SECONDS,  // 时间单位
    workQueue,         // 任务队列
    threadFactory,     // 线程工厂
    handler            // 拒绝策略
);
```

**消息推送避免 OOM：**
- 使用**有界队列**（ArrayBlockingQueue），避免无界队列堆积
- 拒绝策略用 `CallerRunsPolicy`（调用方降速）或自定义（落库重试）
- 核心/最大线程数根据 IO 密集型计算：`线程数 = CPU核数 × (1 + 等待时间/计算时间)`
- 监控队列深度，动态调整参数（用动态线程池框架如 Hippo4j）

### 10. 手写 LRU 缓存（O(1) 读写）

```java
class LRUCache {
    private final int capacity;
    // 双向链表节点
    private final Map<Integer, Node> map = new HashMap<>();
    private final Node head = new Node(0, 0); // 虚拟头
    private final Node tail = new Node(0, 0); // 虚拟尾

    public LRUCache(int capacity) {
        this.capacity = capacity;
        head.next = tail;
        tail.prev = head;
    }

    public int get(int key) {
        if (!map.containsKey(key)) return -1;
        Node node = map.get(key);
        moveToHead(node);  // 最近使用移到头部
        return node.val;
    }

    public void put(int key, int value) {
        if (map.containsKey(key)) {
            Node node = map.get(key);
            node.val = value;
            moveToHead(node);
        } else {
            Node node = new Node(key, value);
            map.put(key, node);
            addToHead(node);
            if (map.size() > capacity) {
                Node lru = removeTail(); // 淘汰最久未使用
                map.remove(lru.key);
            }
        }
    }

    // 核心：HashMap O(1)查找 + 双向链表 O(1)移动
    private void moveToHead(Node node) { removeNode(node); addToHead(node); }
    private void addToHead(Node node) {
        node.prev = head; node.next = head.next;
        head.next.prev = node; head.next = node;
    }
    private void removeNode(Node node) {
        node.prev.next = node.next; node.next.prev = node.prev;
    }
    private Node removeTail() {
        Node node = tail.prev; removeNode(node); return node;
    }

    class Node {
        int key, val;
        Node prev, next;
        Node(int k, int v) { key = k; val = v; }
    }
}
```
**淘汰逻辑：** 链表头 = 最近使用，链表尾 = 最久未使用，每次 get/put 将节点移到头部，容量满时删除尾节点。

### 1. Spring Bean 生命周期 & 三级缓存解决循环依赖

**完整生命周期：**
```
实例化（构造器）
    → 属性填充（依赖注入）
    → Aware 接口回调（BeanNameAware等）
    → BeanPostProcessor#postProcessBeforeInitialization
    → InitializingBean#afterPropertiesSet / init-method
    → BeanPostProcessor#postProcessAfterInitialization
    → 使用
    → DisposableBean#destroy / destroy-method
```

**三级缓存：**

| 缓存 | 存储内容 | 作用 |
|------|---------|------|
| 一级缓存 singletonObjects | 完整Bean | 正常使用 |
| 二级缓存 earlySingletonObjects | 半成品Bean（已实例化未注入） | 解决循环依赖 |
| 三级缓存 singletonFactories | ObjectFactory（Bean工厂） | 支持AOP代理 |

**为什么需要三级？** 若Bean需要AOP代理，二级缓存存的是原始对象，三级缓存通过工厂可以在循环引用时提前返回代理对象，保证注入的是代理。

### 2. 高并发下 DB 与缓存双写一致性

**常见方案对比：**

```
❌ 先写缓存再写DB：DB失败导致缓存脏数据
❌ 先写DB再更新缓存：并发下仍有不一致窗口

✅ Cache-Aside（推荐）：
   写：先写DB → 再删除缓存
   读：缓存miss → 读DB → 写缓存

✅ 延迟双删：
   删缓存 → 写DB → 延迟N毫秒 → 再删缓存
   （消除并发读在写DB期间重建的旧缓存）

✅ 实战方案：
   Canal 订阅 MySQL binlog → 异步删除/更新缓存
   + 消息队列保证删除重试
   + 缓存设置合理TTL兜底
```

### 3. 千万级订单分库分表设计

**分片策略：**
```
分库分表键：order_id（雪花算法生成，包含机器ID+时间戳）
分片规则：order_id % 分库数 → 分库；order_id % 分表数 → 分表
```

**全局唯一 ID：** 雪花算法（64位 = 1符号位 + 41时间戳 + 10机器ID + 12序列号）

**跨库分页：**
```
方案1：禁止深分页，用游标（where id > lastId limit N）
方案2：ES存索引，DB存数据，ES分页后查DB
方案3：业务层归并排序（性能差，不推荐）
```

**扩容问题：**
- 提前按 2^n 分片，扩容时成倍增加，只迁移一半数据
- 双写迁移：新老库同时写，迁移完成后切流量

### 4. MQ 消息丢失、重复消费、堆积解决

**消息丢失：**
```
生产者 → MQ：confirm机制 + 失败重试 + 落库记录
MQ 持久化：消息持久化到磁盘（durable=true）
MQ → 消费者：手动ACK，消费成功再确认
```

**重复消费（幂等）：**
```
方案1：Redis SETNX message_id，处理过直接return
方案2：DB唯一索引，重复插入捕获异常
方案3：乐观锁版本号（支付场景：status=待支付才处理）
```

**消息堆积：**
```
紧急扩容：增加消费者实例数
临时方案：将堆积消息转移到新Topic，扩大消费者
根本原因：排查消费者性能瓶颈（DB慢查询、锁竞争）
实践：限流削峰 + 消费者线程池优化 + 监控告警
```

### 5. 线上 CPU 100%、OOM、死锁、死循环排查

```bash

printf '%x\n' <tid>      # 转16进制
```

```bash
jmap -dump:format=b,file=heap.hprof <pid>
# MAT分析：找GC Roots引用链、大对象、内存泄漏
# 常见原因：集合无限增长、ThreadLocal未remove、大量小对象
```

**死锁：**
```bash

# 自动检测死锁并输出线程持有锁的信息
```

**死循环：**
```
CPU 100% + jstack 发现线程长期在同一位置

```

### 6. 分布式事务 Seata AT/TCC/SAGA & 订单支付落地

**三种模式对比：**

| 模式 | 原理 | 性能 | 适用场景 |
|------|------|------|---------|
| **AT** | 自动补偿，框架生成undo log | 中 | 对业务无侵入，简单场景 |
| **TCC** | Try-Confirm-Cancel，业务自实现 | 高 | 高并发，强一致，支付核心链路 |
| **SAGA** | 长事务，补偿事务链 | 高 | 长流程，跨多服务，最终一致 |

**订单支付 TCC 落地：**
```
Try阶段：
  - 订单服务：冻结库存，订单状态→"待支付"
  - 账户服务：冻结金额

Confirm阶段：
  - 扣减库存，扣减金额，订单状态→"已支付"

Cancel阶段：
  - 释放冻结库存，释放冻结金额，订单→"已取消"

关键点：
  ✅ 幂等：每个阶段用 xid 去重
  ✅ 空回滚：Try未执行，Cancel需正常返回
  ✅ 悬挂：Cancel先于Try到达，Try需拒绝执行
```
