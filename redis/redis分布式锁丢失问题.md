
这是一个经典的分布式系统问题，有几种方案：

## 问题分析

Redis 单节点做分布式锁的核心风险：
- Redis 宕机 → 锁丢失 → 多个进程同时进入临界区
- 主从切换 → 锁未同步到从节点 → 锁失效

---

## 解决方案

### 1. Redlock 算法（Redis 官方推荐）

用 **N 个独立的 Redis 节点**（通常 5 个），加锁时必须在多数节点（≥3）成功才算获锁成功。

**流程：**
1. 记录当前时间戳
2. 依次向 5 个节点请求加锁（每个节点设置相同 key/value/TTL）
3. 若在 **TTL 内** 有 ≥3 个节点加锁成功 → 获锁成功
4. 实际锁有效期 = TTL - 消耗时间
5. 释放时向所有节点发释放命令

**优点：** 单节点故障不影响整体
**缺点：** 实现复杂，性能下降，仍有极端时钟漂移问题（Martin Kleppmann 曾撰文批评）

---

### 2. 锁续期（Watchdog 机制）

不解决宕机，而是**缩短影响窗口**。

```
获锁成功
    ↓
启动后台线程，每 TTL/3 时间自动续期
    ↓
业务完成 → 主动释放锁 + 停止续期
    ↓
若进程崩溃 → 续期停止 → TTL 到期自动释放
```

**Redisson 就是这样实现的**，默认 watchdog 每 10 秒续一次期（锁默认 30 秒过期）。

---

### 3. 持久化 + 快速恢复

配置 Redis AOF 持久化（`appendfsync always`），宕机后快速重启恢复锁状态。

**缺点：** 重启有时间窗口，`always` 模式性能损耗大，不能完全消除风险。

---

### 4. 换用强一致性存储

如果对锁的可靠性要求极高，直接换用基于 **Raft/Paxos** 的系统：

| 方案 | 特点 |
|------|------|
| **ZooKeeper** | 临时节点天然支持锁，会话断开自动释放 |
| **etcd** | lease 机制，强一致，Kubernetes 在用 |
| **Consul** | 支持 session + KV 锁 |

这些系统本身保证多节点一致性，锁的可靠性远高于 Redis。

---

## 如何选择

```
对性能要求高，能接受极小概率风险？
    → 单节点 Redis + Watchdog 续期（Redisson）

需要更高可靠性，还想用 Redis？
    → Redlock（5节点）

对一致性要求严格（金融、库存扣减）？
    → etcd 或 ZooKeeper
```

---

## 核心原则

> 分布式锁本质上无法做到 100% 可靠，业务层面也要做好**幂等处理**，即使锁偶尔失效，重复执行也不会造成数据错误。这才是真正的兜底。
